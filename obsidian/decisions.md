# vehicle_cost_tracker — архітектурні рішення

## Навчальний формат проєкту

Весь код пишеться крок за кроком за `CODING_GUIDE.md` — покроковою
інструкцією для вивчення JS/React/TS з нуля. Кожна фаза = окремий розділ
гайду з поясненнями "навіщо", а не тільки "що робити". Це впливає на темп:
код навмисно пишеться повільніше й послідовніше, ніж міг би.

## Розділення на два репозиторії

Один застосунок (облік транспортних витрат) розділено на:
- `vehicle_cost_tracker` (цей репо) — React PWA фронтенд.
- `vehicle_tracker_api` — Django-бекенд (`C:\Users\b.kysliy\PycharmProjects\vehicle_tracker_api\`).

Причина розділення в контексті не зафіксована окремо — прийнято як
стандартний підхід frontend/backend-репо.

## CSS: Tailwind + окремий CSS-файл для лендінгу

Проєкт має Tailwind у залежностях, але лендінг (`TopNav`, `LandingPage`,
`AuthModal`) написаний на чистому CSS (`src/styles/landing.css`), не
Tailwind-класами. Рішення явно проговорене в `CODING_GUIDE.md` (Крок 3.4)
як усвідомлений вибір для цієї частини — Tailwind лишається для основної
частини застосунку (форми, таблиці, дашборди).

## Auth: cookie, не localStorage

Фаза 4.5 гайду явно обирає cookie-based авторизацію замість localStorage
(Крок 4.5.1 — "Чому cookie, а не localStorage"). Мотивація в гайді:
безпека (XSS не може вкрасти httpOnly cookie) — типове для Django-сесій
рішення, узгоджене з бекендом на `vehicle_tracker_api`.

## `.env`: mock-дані до готовності бекенда

`VITE_USE_MOCK` дозволяє розробляти фронтенд на JSON mock-даних, поки
Django backend (`vehicle_tracker_api`) не готовий або недоступний
локально. `VITE_API_BASE` перемикається між localhost (dev) і
продакшн-доменом (`.env.production`).

## Деплой: власний Raspberry Pi замість хмарного хостингу

Продакшн — домен `warehouse.mom`, що резолвиться на Raspberry Pi вдома,
доступ через Cloudflare Tunnel (`cloudflared`), не прямий порт-форвардинг.
Автодеплой через GitHub Actions: push у `main` → SSH на Pi через тунель →
`git pull` + `docker compose build/up`. Рішення дає повний контроль і
нульову вартість хостингу ціною ручного адміністрування заліза.

## Git-флоу: гілка на фазу, `main` = продакшн

Починаючи з Фази 5 (після налаштування автодеплою у Фазі 4), кожна фаза
розробляється в окремій гілці з PR у `main`, а не напряму. Причина явно
зафіксована в гайді: `main` автоматично деплоїться на `warehouse.mom`,
тому незавершений/зламаний код туди потрапляти не повинен. Фази 1-4
писались напряму в `main`, поки коду було мало і автодеплою ще не було.

## `src/hocks/` — тайпо в назві теки залишено свідомо

Тека хуків названа `hocks` (без другого "o") ще з Фази 3
(`useAuthModal.ts`) — виправлення на `hooks` зараз означало б переписати
всі існуючі імпорти. `CODING_GUIDE.md` (Кроки 4.5.4-4.5.7) у прикладах
коду пише правильне `../../hooks/useCurrentUser` — це розбіжність
гайду з реальною структурою репо, не помилка в коді. При реалізації
нового кроку з гайду копіювати шлях імпорту напряму з тексту гайду
не можна — треба підставляти `hocks`. Саме ця розбіжність спричинила
"файл підкреслений червоним" у `TopNav.tsx` (Крок 4.5.7): імпорт
скопіювали з гайду як `hooks`, а такої теки не існує.
